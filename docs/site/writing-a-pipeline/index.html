Writing a Pipeline

A pipeline is an ordered list of stages built with PipelineBuilder. Stages run head-first in the order you add them, and each stage's emitted frames become the next stage's input.

Build and start

build().start() wires the pipeline and hands back its two ends plus a driver future:

rust 
use pipecrab::{DataFrame, Direction, PipelineBuilder, Received, SystemFrame};

let (ends, driver) = PipelineBuilder::new()
    .stage(ResamplerStage::new(SHERPA_FORMAT)?)  // capture rate → 16 kHz mono
    .stage(VadStage::with_config(detector, cfg)) // gate: emit only utterances
    .stage(SttStage::new(transcriber))           // Audio → Transcript
    .build()
    .start();

let input = ends.input;        // Outbound — feed the head
let mut output = ends.output;  // Inbound  — read past the tail
  • ends.input is an Outbound: everything you send into it enters the head stage.
  • ends.output is an Inbound: everything the tail stage emits comes out of it.
  • driver is the future that actually runs every stage — nothing moves until you await it.

Feed the head, drain the tail

Open the run with a Start system frame, then push data frames. Dropping input closes the head and cascades a clean shutdown downstream.

rust 
let pump_in = async move {
    input.send_system(Direction::Down, SystemFrame::Start).await.ok();
    while let Ok(Some(chunk)) = source.next_chunk().await {
        if input.send_data(DataFrame::Audio(chunk)).await.is_err() {
            break; // downstream gone
        }
    }
    // `input` dropped here → the pipeline shuts down
};

let drain = async move {
    while let Some(received) = output.recv().await {
        if let Received::Data(DataFrame::Transcript(t)) = received {
            println!("{}", t.text);
        }
    }
};

Drive it — on your executor

pipecrab bakes in no executor. The caller runs the future: block_on natively, spawn_local in the browser. Drive the driver and both pumps together on one thread:

rust 
block_on(async { futures::join!(driver, pump_in, drain) });

The pipeline is one !Send thread by design. Heavy work inside stages crosses to a std::thread (native) or a Web Worker (wasm) via offload — see Writing a Stage.

The two lanes

Every link between stages carries two queues:

LaneCarriesDirectionPriority
Data DataFrame — Audio, Transcript, SpeechStarted/SpeechStopped, InputAudio, Custom downstream only normal
System SystemFrame — Start, Stop, Interrupt, Error both directions drained first; can preempt in-flight work

The system lane is only for frames that are allowed to overtake. State changes like speech edges ride the data lane, in order with the audio they bracket. DataFrame and SystemFrame are distinct types with no shared inheritance tree, so you can't accidentally push a system frame into the downstream-only data lane.

On an Interrupt, queued data frames are flushed — except survivors: input audio from the transport survives the flush so a barge-in utterance is never clipped.

Nesting and backpressure

A Pipeline is itself a Stage, so a whole pipeline can be passed to .stage(..) to nest it inside another. PipelineBuilder::capacity(n) sets the per-lane buffer depth — how many frames may queue on a lane before a send awaits (backpressure).

rust 
let transcribe = PipelineBuilder::new()
    .stage(VadStage::with_config(detector, cfg))
    .stage(SttStage::new(transcriber))
    .build();

let (ends, driver) = PipelineBuilder::new()
    .capacity(32)
    .stage(ResamplerStage::new(SHERPA_FORMAT)?)
    .stage(transcribe)   // a pipeline nested as one stage
    .build()
    .start();

For a complete, runnable version of the pipeline on this page, see examples/stt-sherpa; for the full duplex loop, the Quick Start.
